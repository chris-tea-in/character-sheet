## ✅ 🔴 BUG-01 — Feat removal does not revert ASI, speed, initiative, or save proficiency

**Files:** `src/components/sheet/FeatsBlock.tsx:237-248`

**Description:**
`removeFeat()` deletes `featChoices[key]` and adjusts `maxHp` (for Tough feat) and `currentHp`, but it never reverts ability score changes, speed bonuses, initiative bonuses, or save proficiencies that the feat granted when it was added.

Concrete example: A player adds the Resilient (CON) feat during character creation or level-up. CON is incremented by +1 in `character.abilities`. Later they remove the feat from `FeatsBlock`. The +1 CON persists permanently in the database.

```ts
// FeatsBlock.tsx removeFeat() — only handles HP; nothing else reverted
function removeFeat(key: string) {
  const newFeatChoices = { ...character.featChoices }
  delete newFeatChoices[key]
  const newFeats = character.feats.filter(f => f !== key)
  const newAdjustedMax = character.maxHp + computeFeatHpBonus(newFeats, character.level)
  onSave({
    feats: newFeats,
    featChoices: newFeatChoices,
    currentHp: Math.min(character.currentHp, newAdjustedMax),
    // ❌ abilities NOT reverted
    // ❌ speed NOT reverted
    // ❌ initiativeBonus NOT reverted
    // ❌ savingThrowProficiencies NOT reverted
  })
}
```

**Fix:**
Before removing the feat, compute its delta via `computeFeatStatDelta`, then:
- `unapplyFeatAsi(character.abilities, delta.abilities)` and save the result
- `character.speed - delta.speed` if `delta.speed !== 0`
- `character.initiativeBonus - delta.initiativeBonus` if `delta.initiativeBonus !== 0`
- Remove `delta.saveProficiency` from `savingThrowProficiencies` if it was feat-granted

**Fix applied (2026-06-12, fb43b5b):** FeatsBlock.removeFeat no longer needs to revert anything — feat ASI/speed/initiative/save effects are derived at render time from `feats` + `featChoices`, so deleting the feat removes them automatically. Only the HP clamp remains in removeFeat.

## ✅ 🔴 BUG-02 — Feat speed and initiative bonuses are double-counted

**Files:** `src/lib/characterSetup.ts:659,662` · `src/lib/characterStats.ts:489-491`

**Description:**
`draftToNewCharacter` bakes feat speed and initiative bonuses into the stored character record:

```ts
// characterSetup.ts draftToNewCharacter
speed: (subraceData?.speed ?? race?.base.speed ?? 30) + featSpeedBonus,
initiativeBonus: featInitiativeBonus,
```

`deriveCharacterStats` then applies the same feat bonuses a second time:

```ts
// characterStats.ts deriveCharacterStats
const effectiveSpeed = character.speed + featSpeedBonus          // double-counted
const effectiveInitiativeBonus = (character.initiativeBonus ?? 0) + featInitiativeBonus  // double-counted
```

Any character created via the setup wizard with a speed-affecting feat (e.g. Squat Nimbleness +5) or initiative feat (e.g. Alert +5) will display incorrect stats. A human with Squat Nimbleness would show 40 ft speed instead of 35 ft, and an Alert character would show +5 extra initiative bonus.

`LevelUpDialog.handleApply` has the same issue — it bakes speed/initiative into the stored record when a feat is applied at level-up, and then `deriveCharacterStats` adds it again.

**Fix:**
Adopt a consistent policy — render-time derivation is the right approach (already used for everything else). Remove the feat bonuses from `draftToNewCharacter`:

```ts
speed: subraceData?.speed ?? race?.base.speed ?? 30,  // base only
initiativeBonus: 0,                                    // base only
```

And remove the bake-in from `LevelUpDialog.handleApply`:

```ts
// Remove these lines from handleApply:
if (delta.speed !== 0)
  changes.speed = character.speed + delta.speed
if (delta.initiativeBonus !== 0)
  changes.initiativeBonus = (character.initiativeBonus ?? 0) + delta.initiativeBonus
```

`deriveCharacterStats` already handles both correctly via `featSpeedBonus` and `featInitiativeBonus`.

**Fix applied (2026-06-12, fb43b5b):** Write-time baking removed at every site: `draftToNewCharacter` stores base speed and `initiativeBonus: 0`; the `LevelUpDialog.handleApply` feat branch writes only `feats` + `featChoices`. `deriveCharacterStats` is the single application point. (Feat ASIs turned out to be double-counted too, not just speed/initiative — same fix covers them.)

## ✅ 🔴 BUG-05 — `unapplyFeatAsi` reduces ability below pre-feat value when bonus was capped

**Files:** `src/lib/characterStats.ts:105-113`

**Description:**
`applyFeatAsi` caps at 20 but does not record how much was actually applied. `unapplyFeatAsi` blindly subtracts the full `amount` regardless of how much was actually applied:

```ts
// applyFeatAsi — may cap and apply less than `amount`
result[ab] = Math.min(20, result[ab] + amount)

// unapplyFeatAsi — always subtracts full `amount`
result[ab] = result[ab] - amount   // ❌ no floor, no tracking of actual delta
```

Scenario: Player has STR 20 (naturally), adds a feat that grants +1 STR. `applyFeatAsi` caps at 20 — no change to the stored value. `unapplyFeatAsi` then subtracts 1 → STR becomes 19, which is wrong.

This bug is currently dormant because `FeatsBlock.removeFeat` never calls `unapplyFeatAsi` (BUG-01). When BUG-01 is fixed, BUG-05 will trigger in this scenario.

**Fix:**
Track what was actually applied:
```ts
export function applyFeatAsi(abilities: Abilities, delta: Partial<Record<AbilityName, number>>): { abilities: Abilities; applied: Partial<Record<AbilityName, number>> } {
  const result = { ...abilities }
  const applied: Partial<Record<AbilityName, number>> = {}
  for (const [ab, amount] of Object.entries(delta) as [AbilityName, number][]) {
    const before = result[ab]
    result[ab] = Math.min(20, result[ab] + amount)
    applied[ab] = result[ab] - before   // actual change
  }
  return { abilities: result, applied }
}
```

Then `unapplyFeatAsi` should subtract only the stored `applied` amount, not the original `amount`.

**Fix applied (2026-06-12, fb43b5b):** `applyFeatAsi`/`unapplyFeatAsi` deleted — feat ASIs are never baked into stored abilities, so there is nothing to un-apply; the 20-cap lives only in the render-time derivation.

## ✅ 🔴 BUG-13 — Edit flow re-applies racial ASIs on every save (permanent ability inflation)

**Files:** `src/lib/characterSetup.ts:701-759` (characterToDraft) · `src/lib/characterSetup.ts:566` (draftToNewCharacter)

**Description:**
`characterToDraft` copies the character's **stored** abilities into the draft — these already include racial/subrace ASIs baked in at creation:

```ts
abilities: { ...character.abilities },   // already includes +2 CON etc. from race
```

`draftToNewCharacter` then **unconditionally** re-applies racial bonuses on save:

```ts
const racialAbilities = applyRaceAsi(draft.abilities, race, draft.asiChoices, draft.subraceSlug)
```

`applyRaceAsi` applies `race.base.ability_score_increases` and subrace ASIs regardless of `abilityMethod`. So every Edit → Save cycle adds the racial bonuses **again**.

Concrete example: A Hill Dwarf with CON 16 (14 base + 2 racial) opens the edit wizard and clicks "Save Changes" without touching anything. CON becomes 18. Edit again → 20. The corruption is permanent and compounds with each edit.

**Fix:**
Either (a) store base abilities in the draft by *subtracting* racial bonuses in `characterToDraft`:
```ts
const racialBonuses = getRacialBonuses(race, [], character.subrace ?? undefined)
abilities: subtractBonuses(character.abilities, racialBonuses),
```
or (b) add a draft flag (`abilitiesAreFinal: true`) set by `characterToDraft` that makes `draftToNewCharacter` skip `applyRaceAsi` (and skip class-ASI application) when editing. Option (b) is safer — it avoids guessing which flexible ASI choices (`asiChoices`) were originally made, which cannot be reconstructed from the stored record.

**Fix applied (2026-06-12, fb43b5b + 9613e8a):** Abilities are stored as BASE scores and racial ASIs are derived at render time, so there is nothing to re-apply: `draftToNewCharacter` no longer calls `applyRaceAsi`, and `characterToDraft` round-trips `raceAsiChoices` and sets `editMode`. Edit → Save is idempotent.

## 🟡 BUG-28 — Level-up ASI cannot put +2 into a single ability, despite the UI saying it can

**Files:** `src/components/sheet/LevelUpDialog.tsx:170-179, 465-466, 272-275`

**Description:**
The ASI help text reads "Choose up to 2 abilities to increase by +1 each **(or the same ability twice for +2)**", and the buttons render `+${count}` as if count could reach 2. But `toggleAsi` treats any second click on an already-selected ability as a *deselect*:

```ts

  if (asiChoices.filter(x => x === ab).length > 0) {
    // ❌ already selected once → REMOVE it, instead of adding the second +1
    setAsiChoices(c => { const idx = c.lastIndexOf(ab); return c.filter((_, i) => i !== idx) })
  } else if (asiChoices.length < 2) {
    setAsiChoices(c => [...c, ab])

}
```

**Fix:**
Allow a second click to stack while under the cap, and provide an explicit removal path:
```ts
function toggleAsi(ab: AbilityName) {
  const count = asiChoices.filter(x => x === ab).length
  if (count === 1 && asiChoices.length < 2) {
    setAsiChoices(c => [...c, ab])          // second click → +2
  } else if (count > 0) {
    setAsiChoices(c => c.filter(x => x !== ab))  // third click → clear both
  } else if (asiChoices.length < 2) {
    setAsiChoices(c => [...c, ab])
  }
}
```
Also guard the `+2` path with `character.abilities[ab] + count < 20` (already partially handled by the `maxed` check).
